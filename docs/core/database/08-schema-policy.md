# database тАФ SQL schema layout policy

> Where each kind of SQL file lives, when to write to which folder, and the naming convention. The runtime side (how it gets applied) is in [`07-update-mechanism.md`](./07-update-mechanism.md). For a full table inventory of what exists in each DB, see [`../../09-db-tables.md`](../../09-db-tables.md).

|---|---|

## Key concepts

- **Three-layer model** тАФ
  1. **`base/db_<name>/`** is the *snapshot*. It is regenerated periodically from the production-shape DB and committed by maintainers (CI warning + maintainer approval if a PR touches it). Never edit by hand.
  2. **`updates/db_<name>/`** is the *append-only* migration log applied on top of `base/`. Every file is identified by its filename; sort order is the apply order. State `RELEASED`.
  3. **`updates/pending_db_<name>/`** is the *PR staging area*. Files have throwaway names (`rev_<unix-nanoseconds>.sql`) so PRs don't conflict on filename. State `PENDING` while there. On merge, a maintainer renames to `YYYY_MM_DD_NN.sql` and moves to `updates/db_<name>/` (the script is otherwise unchanged).
- **Archive policy** тАФ periodically the maintainers "squash" old migrations: their cumulative effect is folded back into `base/` and their files are moved to `archive/db_<name>/`. The corresponding rows in the `updates` table get `state='ARCHIVED'` (left in place so deployments that already ran them don't re-trigger). See `data/sql/updates/database-squash.md`.
- **Naming convention for migrations** тАФ
  - `YYYY_MM_DD_NN.sql` (e.g. `2025_01_05_03.sql`). Day-counter `NN` resets per day, two digits, zero-padded.
  - Generated by `pending_db_*/create_sql.sh`: `rev_<seconds><ns>.sql` to avoid PR conflicts.
- **Re-applicable scripts** тАФ files that must run on every boot when their content changes (e.g. updating fixed enum tables) go under `data/sql/custom/db_<name>/`. The README enforces idempotency: `CREATE TABLE IF NOT EXISTS`, `REPLACE INTO`, `DELETE+INSERT`, fixed-value `UPDATE`. State `CUSTOM`.
- **Module schemas** тАФ modules ship their SQL under `modules/<name>/data/sql/db-<name>/` (note the dash, not underscore). They register the directory via the module's `include.sh` (sourced by `apps/bash_shared/common.sh:22`), which inserts a row into `updates_include` with `state='MODULE'`. Files use the same `YYYY_MM_DD_NN.sql` convention. State `MODULE`.

## Flow / data shape

### Decision tree тАФ "I need to change the DB schema"

```
Are you adding a new column / table / row that should ship with HEAD?
   тФЬтФА Yes
   тФВ   тФЬтФА Is it part of a module?
   тФВ   тФВ   тФЬтФА Yes тЖТ modules/<mod>/data/sql/db-world/ (or db-characters/)
   тФВ   тФВ   тФВ       file:  YYYY_MM_DD_NN.sql      state: MODULE
   тФВ   тФВ   тФФтФА No тЖТ data/sql/updates/pending_db_<name>/
   тФВ   тФВ           file:  rev_<ns>.sql via create_sql.sh   state: PENDING
   тФВ   тФВ           (renamed to YYYY_MM_DD_NN.sql by maintainer on merge)
   тФВ   тФФтФА End. Never touch data/sql/base/.
   тФВ
   тФФтФА No, it's a fix you want re-applied whenever its content changes
       (idempotent statements only тАФ see custom/README.md)
       тФФтФА data/sql/custom/db_<name>/  state: CUSTOM
```

### Folder summary

| Folder | Edited by | State in `updates` table | Purpose |
|---|---|---|---|
| `data/sql/base/db_<name>/` | Maintainers (squash routine) | n/a тАФ applied via `Populate`, no row | Initial snapshot |
| `data/sql/updates/db_<name>/` | Maintainers (post-merge) | `RELEASED` | Stable migrations |
| `data/sql/updates/pending_db_<name>/` | PR authors | `PENDING` | PR staging |
| `data/sql/archive/db_<name>/` | Maintainers (squash routine) | `ARCHIVED` | Historical, hash kept |
| `data/sql/custom/db_<name>/` | PR authors (re-applicable only) | `CUSTOM` | Fix loops |
| `data/sql/old/db_<name>/` | Historical | not tracked | Not consumed |
| `data/sql/create/` | Maintainers | not tracked | Bootstrap scripts |
| `modules/<m>/data/sql/db-<name>/` | Module author | `MODULE` | Module schema |

### File header convention

Every SQL file should start with the standard guard block (no requirement enforced by the fetcher, but uniform across the tree). The CI codestyle script `apps/codestyle/codestyle-sql.py` (see [`../../07-codestyle.md`](../../07-codestyle.md)) enforces formatting.

## Hooks & extension points

- **Module authors** тАФ drop SQL into `modules/<m>/data/sql/db-<world|characters|auth>/` and ensure the module's `include.sh` adds the directory to `updates_include`. Convention: keep one file per concern, name it `YYYY_MM_DD_NN.sql`.
- **PR authors** тАФ never touch `base/` (CI will yell). Add migrations only in `pending_db_<name>/` using `create_sql.sh`. The PR review will rename the file before merge.
- **Maintainers** тАФ periodic "squash" cycles condense `updates/db_<name>/`'s oldest files into a refreshed `base/db_<name>/` dump; the squashed files move to `archive/db_<name>/`. Document the boundary in `database-squash.md`.

# scripting — Module discovery, loader generation, naming convention

> A module is *any* directory under `modules/` that contains a `src/` subdirectory. CMake globs them, generates one C++ aggregator per module (or one big one for static linkage), and `worldserver` invokes the aggregator at boot via `_modules_loader_callback` (see [`01-script-mgr.md`](./01-script-mgr.md)). For the static/dynamic linkage modes see [`06-static-vs-dynamic.md`](./06-static-vs-dynamic.md). Per-module catalog: [`../../05-modules.md`](../../05-modules.md).

## Key concepts

- **Module = a directory** — `modules/<name>/src/` is the only required convention. An optional `<name>/CMakeLists.txt` lets a module add custom sources/configs; `<name>/<name>.cmake` lets a module run inline at configure time (`modules/CMakeLists.txt:301-303`).
- **Loader function naming** — `Add<module-name-with-dashes-replaced-by-underscores>Scripts()`. Generated by the regex at `modules/CMakeLists.txt:153` (`string(REGEX REPLACE - "_" ...)`). Examples:
  - `mod-paragon` → `Addmod_paragonScripts()`
  - `mod-paragon-itemgen` → `Addmod_paragon_itemgenScripts()`
  - `mod-loot-filter` → `Addmod_loot_filterScripts()`
  - `mod-endless-storage` → `Addmod_endless_storageScripts()`
  - `mod-custom-spells` → `Addmod_custom_spellsScripts()`
  - **Special case** `mod-auto-loot` → `AddSC_AutoLoot()` (legacy `AC_ADD_SCRIPTS_LIST` API path; see `modules/CMakeLists.txt:32-46`, `198-205`). The loader generator also accepts that form.
- **`AC_ADD_SCRIPTS_LIST` / `AC_ADD_SCRIPTS_INCLUDE`** (deprecated) — alternative API where a module pushes onto two CMake lists at configure time. Triggered when a module's loader cpp is referenced via `AC_SCRIPTS_INCLUDES`. Forces static linkage (line 75-77). New modules should use the auto-detected `Add<modulename>Scripts()` form instead.
- **Disabling a module** — `cmake -DDISABLED_AC_MODULES="mod-foo;mod-bar"` (semicolon-separated list). Filtered at `CMakeLists.txt:77`. Disabled-module shared libraries are also explicitly removed from the install dir (`modules/CMakeLists.txt:317-327`).
- **Module configs** — `<module>/conf/*.conf.dist` are auto-collected into `CONFIG_FILE_LIST` (lines 342-353) and copied next to `worldserver.conf.dist` at install time.
- **`-D<MODULE_NAME>=`** — every module gets a per-module CMake cache variable (`MODULE_MOD_PARAGON=default|disabled|static|dynamic`, see `conf/dist/config.cmake:74-79`). `default` follows the global `-DMODULES=` setting.

## Flow / data shape

```
Configure time (cmake ..)
  CMakeLists.txt
    └─ CU_SUBDIRLIST(.../modules)
        └─ for each: skip if in DISABLED_AC_MODULES, else add_subdirectory()
  modules/CMakeLists.txt
    ├─ GetModuleSourceList()                  # globs */src
    ├─ for each module: pick linkage (default | static | dynamic | disabled)
    ├─ for static modules: collect sources -> umbrella libmodules
    ├─ for dynamic modules: add_library(mod_<name> SHARED ...)
    │     ConfigureScriptLoader(<name>, ..., IS_DYNAMIC=ON, <name>)
    │       → generates gen_scriptloader/<name>/ModulesLoader.cpp
    │         containing extern "C" { GetScriptModule(); AddModulesScripts(); }
    └─ ConfigureScriptLoader("static", ..., IS_DYNAMIC=OFF, <static_modules...>)
          → generates gen_scriptloader/static/ModulesLoader.cpp
            containing forward decls + AddModulesScripts() that calls every Add*Scripts()

At server start (worldserver Main.cpp:266)
  sScriptMgr->SetModulesLoader(&AddModulesScripts);   // statically-linked symbol
  sScriptMgr->Initialize();
    └─ _modules_loader_callback();                    // → AddModulesScripts()
        ├─ Addmod_paragonScripts();
        ├─ Addmod_loot_filterScripts();
        ├─ AddSC_AutoLoot();                          // legacy form, see special case
        └─ ... (one call per static module; dynamic modules expose their own AddModulesScripts via dlopen path)
```

The generated `ModulesLoader.cpp` (template at `modules/ModulesLoader.cpp.in.cmake:49`) is identical in shape for static vs. dynamic; the only difference is whether `extern "C"` wraps it (line 34 / 62). The dynamic case also exports `GetScriptModule()` so `worldserver` can identify the loaded `.so`/`.dll` at runtime.

## Hooks & extension points

- **Add a module** — drop it under `modules/<name>/`, ensure `modules/<name>/src/` exists, define `void Add<sanitized_name>Scripts()` somewhere in those sources, and re-run `cmake ..`. Wiki: `wiki/Create-a-Module`.
- **Per-module linkage override** — `cmake .. -DMODULE_MOD_FOO=dynamic` overrides the global setting just for `mod-foo`.
- **Per-module CMake** — adding `modules/<name>/CMakeLists.txt` lets you fetch deps, set defines, etc. Adding `modules/<name>/<name>.cmake` runs inline at the parent `modules/CMakeLists.txt` scope (`modules/CMakeLists.txt:301-303`).
